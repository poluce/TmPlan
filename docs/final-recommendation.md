# TmPlan 综合调研报告 — 最终推荐方案（修订版）

> 本报告基于团队调研 + 用户深度访谈，已根据用户真实需求修订。

## 一、产品定位与核心价值

### 一句话定位
**TmPlan — 面向开发者的AI引导式计划文件生成器，把模糊想法变成AI可执行的精细计划。**

### 核心理念
TmPlan不是项目管理工具，不是代码编辑器。它只做一件事：
**通过引导式选择，帮开发者生成颗粒度极细的计划文件，细到任何主流AI模型拿到就能直接执行。**

### 解决的核心痛点

| 痛点 | 具体表现 | TmPlan如何解决 |
|------|---------|---------------|
| AI上下文有限 | 每次开新对话都要重新讲背景，AI经常"忘记"之前的决策 | 所有决策记录在计划文件中，AI每次只需读文件 |
| 计划与代码脱节 | 计划写了一版，实际开发偏离了，没人维护一致性 | AI自动检测代码变化与计划的偏差，给出选择题让用户决策 |
| 想法模糊难落地 | 用户有感觉但说不清楚，AI一次性生成的计划太笼统 | 不一次性生成，而是通过多轮选择题逐步引导，每一步都是用户确认的 |
| 新增模块难集成 | 想加功能但不知道怎么融入现有架构，AI给的方案可能冲突 | TmPlan读取现有代码结构，基于实际情况给出不冲突的选择 |

### 目标用户
- **使用AI辅助开发的程序员** — 用Claude Code、Cursor、GitHub Copilot等工具写代码
- **独立开发者/小团队** — 需要管理项目计划但不想用重型工具
- **对AI开发有痛点的人** — 经历过"AI上下文丢失"、"计划与代码冲突"的困扰

### 与竞品的本质区别

| 维度 | Jira/Linear | Notion | AI编码工具 | TmPlan |
|------|------------|--------|-----------|--------|
| 定位 | 任务管理 | 万能工作空间 | 写代码 | 生成计划文件 |
| 计划创建 | 手动填写 | 手动/AI辅助 | 不管计划 | AI引导式选择 |
| 计划颗粒度 | 粗（标题+描述） | 中（自由格式） | 无 | 极细（AI可直接执行） |
| 与代码的关系 | 不看代码 | 不看代码 | 只看代码 | 只读代码，只写计划 |
| 冲突检测 | 无 | 无 | 无 | AI自动检测计划vs代码偏差 |

## 二、核心用户场景

### 场景1：从模糊想法到精细计划（引导式决策树）

TmPlan不会一次性生成计划，而是像一个资深顾问一样，通过选择题+对话逐步引导：

```
用户："我想做一个博客系统"

TmPlan：好的，我们来一步步理清——

  Q1: 这个博客主要给谁用？
      [A] 个人博客（自己写文章）
      [B] 多人博客平台（多作者）
      [C] 企业内容管理
      → 用户选 [A]

  Q2: 你希望用什么技术栈？
      [A] Next.js + React（SSR，SEO友好）
      [B] Nuxt.js + Vue（类似方案，Vue生态）
      [C] 纯静态生成（Astro/Hugo，最简单）
      [D] 不确定，帮我分析
      → 用户选 [A]

  Q3: 内容存储方式？
      [A] Markdown文件（简单，Git管理）
      [B] 数据库（灵活，支持在线编辑）
      [C] CMS（Strapi/Sanity，功能全但复杂）
      → 用户选 [B]

  Q4: 需要哪些功能？（多选）
      [A] 文章CRUD
      [B] 标签/分类
      [C] 评论系统
      [D] 搜索
      [E] RSS订阅
      → 用户选 [A][B][D]

  Q5: 你提到了搜索，搜索方案——
      [A] 数据库全文搜索（简单，够用）
      [B] Algolia/MeiliSearch（体验好，有成本）
      → 用户选 [A]

  ... 继续从部署、样式、SEO等维度引导 ...
```

每一步选择都会影响后续的选项分支。选了A可能跳到D，选了B可能跳到E。
最终输出的计划是用户一步步"选"出来的，不是AI凭空生成的。

### 场景2：计划与代码冲突检测

TmPlan和编码工具（Claude Code、VS Code等）同时打开同一个项目：

```
┌──────────────────┐    ┌──────────────────┐
│  编码工具         │    │    TmPlan        │
│  (写代码)         │    │   (看计划)        │
│                  │    │                  │
│  正在开发认证模块  │    │  ┌────────────┐  │
│                  │    │  │项目总进度 45%│  │
│                  │    │  └────────────┘  │
│                  │    │  认证模块 🔄 开发中│
│                  │    │  博客CRUD ⏳ 待开始│
│                  │    │  搜索功能 ⏳ 待开始│
│                  │    │                  │
│                  │    │  [手动检查] 按钮   │
└────┬─────────────┘    └──────┬───────────┘
     └──── 同一个项目目录 ──────┘
```

用户点击"手动检查"或到了定时检查时间（如每5分钟）：

```
TmPlan检测到偏差：

  ⚠️ 认证模块 — 计划vs实际不一致

  计划：使用 Supabase Auth
  实际：代码中引入了 next-auth 包

  这会影响：
  - 数据库schema需要调整（原计划用Supabase的users表）
  - 后续的"用户设置"模块也需要改

  请选择：
  [A] 更新计划为 NextAuth.js（AI将自动调整相关模块的计划）
  [B] 回退代码，继续用 Supabase Auth
  [C] 让我详细说明为什么改了（进入对话模式）

  → 用户选 [A]
  → TmPlan自动更新计划文件，所有受影响的模块计划同步调整
```

### 场景3：新增模块

```
用户："我想加一个评论功能"

TmPlan不会直接生成方案，而是先读取现有代码结构，然后引导：

  TmPlan：我看了你的项目结构，目前有以下情况——
  - 数据库：Supabase PostgreSQL
  - 认证：NextAuth.js
  - 现有API：/api/posts, /api/tags

  Q1: 评论系统的方案？
      [A] 自建评论（存自己数据库，完全可控）
      [B] 第三方服务（Disqus/Giscus，省事但依赖外部）
      → 基于你现有的Supabase，我建议 [A]，可以复用现有数据库
      → 用户选 [A]

  Q2: 评论需要支持嵌套回复吗？
      [A] 不需要，扁平评论就够
      [B] 需要，支持一级回复
      [C] 需要，支持多级嵌套
      → 用户选 [B]

  Q3: 评论审核机制？
      [A] 无审核，直接发布
      [B] 先审后发
      [C] AI自动过滤垃圾评论 + 人工审核
      → 用户选 [A]

  → TmPlan生成评论模块的计划文件，自动插入到项目计划中
  → 看板上出现新的"评论模块"卡片
  → 自动计算对现有进度的影响
```

## 三、计划文件体系

### 核心原则
计划文件的颗粒度要细到：**任何主流AI模型拿到这份文件，就能直接执行对应的开发任务。**

### 文件结构

```
项目根目录/
└── .tmplan/                        # TmPlan计划文件目录
    ├── project.yaml                # 项目总览
    ├── decisions/                  # 决策记录（每一步选择的记录）
    │   ├── 001-target-user.yaml
    │   ├── 002-tech-stack.yaml
    │   ├── 003-storage.yaml
    │   └── ...
    ├── modules/                    # 模块级计划（AI可直接执行的颗粒度）
    │   ├── auth.yaml
    │   ├── blog-crud.yaml
    │   ├── search.yaml
    │   └── ...
    ├── phases/                     # 阶段划分
    │   ├── phase-1-mvp.yaml
    │   └── phase-2-enhance.yaml
    └── status.yaml                 # 当前状态（进度、偏差、冲突）
```

### 模块计划文件示例

以认证模块为例，颗粒度细到AI可以直接执行：

```yaml
# .tmplan/modules/auth.yaml
module: 用户认证
status: pending
depends_on: []
decision_ref: decisions/003-auth-method.yaml

overview: |
  使用NextAuth.js实现GitHub OAuth登录，
  用户数据存储在Supabase PostgreSQL中。

tasks:
  - id: auth-01
    title: 安装和配置NextAuth.js
    status: pending
    detail: |
      1. 安装依赖：next-auth@5 @auth/supabase-adapter
      2. 创建 app/api/auth/[...nextauth]/route.ts
         - 配置 GitHub OAuth Provider
         - 配置 Supabase Adapter
      3. 创建 lib/auth.ts 导出 auth 配置
      4. 在 .env.local 中添加：
         - GITHUB_ID=xxx
         - GITHUB_SECRET=xxx
         - NEXTAUTH_SECRET=xxx（用openssl rand -base64 32生成）
    files_to_create:
      - app/api/auth/[...nextauth]/route.ts
      - lib/auth.ts
    files_to_modify: []
    acceptance_criteria:
      - 访问 /api/auth/signin 显示GitHub登录按钮
      - 点击后跳转GitHub授权页面
      - 授权后回调到应用，session中包含用户信息

  - id: auth-02
    title: 创建用户数据表
    status: pending
    depends_on: [auth-01]
    detail: |
      在Supabase中创建迁移文件：
      supabase/migrations/001_create_auth_tables.sql

      表结构：
      - users: id(uuid PK), name(text), email(text UNIQUE),
               email_verified(timestamptz), image(text),
               created_at(timestamptz DEFAULT now())
      - accounts: id(uuid PK), user_id(uuid FK->users),
                  type(text), provider(text),
                  provider_account_id(text), ...
      - sessions: id(uuid PK), session_token(text UNIQUE),
                  user_id(uuid FK->users), expires(timestamptz)

      RLS策略：
      - users: 用户只能SELECT/UPDATE自己的记录
    files_to_create:
      - supabase/migrations/001_create_auth_tables.sql
    acceptance_criteria:
      - 运行 supabase db push 后表创建成功
      - 用户登录后自动在users表创建记录

  - id: auth-03
    title: 实现登录/登出UI
    status: pending
    depends_on: [auth-01, auth-02]
    detail: |
      1. 创建 components/auth/sign-in-button.tsx
         - 未登录显示"Sign in with GitHub"按钮
         - 已登录显示用户头像和名字
      2. 创建 components/auth/user-menu.tsx
         - 下拉菜单：个人设置、登出
      3. 在顶部导航栏集成登录组件
      4. 创建 middleware.ts 保护需要登录的路由
    files_to_create:
      - components/auth/sign-in-button.tsx
      - components/auth/user-menu.tsx
      - middleware.ts
    acceptance_criteria:
      - 未登录用户访问/dashboard被重定向到登录页
      - 登录后显示用户头像，点击可登出
```

### 决策记录文件示例

```yaml
# .tmplan/decisions/003-auth-method.yaml
decision_id: 003
question: "认证方式选择"
context: |
  项目使用Next.js + Supabase，需要用户登录功能。
options_presented:
  - id: A
    label: Supabase Auth
    description: Supabase内置认证，配置简单
  - id: B
    label: NextAuth.js
    description: 灵活，支持多种Provider，社区活跃
  - id: C
    label: Clerk
    description: 托管认证服务，开箱即用但有费用
chosen: B
reason: "用户选择NextAuth.js，因为需要自定义OAuth Provider的灵活性"
impact:
  - 需要自建用户表（不用Supabase Auth的内置表）
  - 后续模块的用户身份获取方式：通过NextAuth session
decided_at: "2026-02-14"
```

## 四、产品形态与使用方式

### 产品形态
**先Web后桌面** — 先做Web版快速验证，后期用Tauri打包成轻量桌面应用。

### 使用方式
TmPlan和编码工具**并排使用**，同时打开同一个项目：

```
用户的屏幕：
┌─────────────────────────┬─────────────────────────┐
│                         │                         │
│     编码工具（左半屏）    │     TmPlan（右半屏）     │
│                         │                         │
│  Claude Code / VS Code  │  浏览器打开TmPlan        │
│  Cursor / 其他IDE       │  指向同一个项目目录       │
│                         │                         │
│  → 写代码、改代码        │  → 看进度、做决策        │
│                         │                         │
└─────────────────────────┴─────────────────────────┘
```

### 检查机制（计划vs代码一致性）

| 方式 | 触发条件 | 说明 |
|------|---------|------|
| 手动检查按钮 | 用户点击界面上的"检查"按钮 | 最常用，用户主动触发 |
| 定时检查 | 用户设定间隔（1分钟/5分钟/自定义） | 可选，自动但节省token |
| Git commit检查 | 每次git commit时自动触发 | 通过Git Hook集成 |
| CLI命令 | 终端运行 `tmplan check` | 开发者友好，可集成到脚本 |

检查时TmPlan会：
1. 只读扫描项目代码结构（不改代码）
2. 对比计划文件中的预期 vs 实际代码
3. 发现偏差时在看板上标注，并给出选择题让用户决策

## 五、界面设计

### 核心布局：两层流水线

页面分为**上下两个区域**，上面是第一层（模块/步骤级），下面是第二层（子任务级）。
每一层都是从左到右排列，体现执行顺序和依赖关系。

```
┌──────────────────────────────────────────────────────────────┐
│  TmPlan    [项目名称 ▼]         [手动检查]  [+ 新增模块]  [设置] │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  第一层：模块/步骤总览（从左到右 = 执行顺序）                    │
│                                                              │
│  ┌────────┐   ┌────────┐   ┌────────┐ ┌────────┐            │
│  │ 认证模块 │──→│ 博客CRUD│──→│ 搜索功能 │ │ 评论功能 │           │
│  │  ✅ 完成 │   │ 🔄 进行 │   │ ⏳ 待开始│ │ ⏳ 待开始│           │
│  │  3/3   │   │  2/5   │   │  0/3   │ │  0/4   │           │
│  └────────┘   └────────┘   └────────┘ └────────┘            │
│                    ↑                    ↑                     │
│                 当前选中             这两个可以并行              │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  第二层：选中模块的子任务（从左到右 = 执行顺序）                  │
│  ── 博客CRUD模块 ──                                           │
│                                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│  │创建数据模型│─→│实现API路由│─→│文章列表页面│  │文章编辑器  │     │
│  │  ✅ 完成  │  │  ✅ 完成  │  │  🔄 进行中│  │  ⏳ 待开始 │     │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘     │
│                                    ↑        ↑                │
│                                 有先后关系    这两个可以并行     │
│                                                              │
│  ┌──────────┐                                                │
│  │文章详情页面│                                                │
│  │  ⏳ 待开始 │                                                │
│  └──────────┘                                                │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 布局规则

**从左到右 = 执行顺序/依赖关系**
- 左边的必须先完成，右边的才能开始
- 如果两个卡片在同一列（上下排列），表示它们**可以并行**执行
- 用箭头（→）连接有依赖关系的卡片

**上下两层联动**
- 第一层：展示所有模块/大步骤，点击某个模块
- 第二层：展示该模块内的所有子任务，布局规则和第一层一样（左到右有序，同列可并行）

**示意图（依赖 + 并行）**

```
第一层示例：有依赖的串行 + 可并行的并排

  ┌────┐   ┌────┐   ┌────┐
  │ A  │──→│ B  │──→│ D  │
  └────┘   └────┘   └────┘
                     ┌────┐
                     │ E  │    ← D和E可以并行，所以同列上下排
                     └────┘

  含义：A必须先完成 → 然后B → 然后D和E可以同时做

第二层示例：点击B模块后展开

  ┌──────┐   ┌──────┐   ┌──────┐
  │ B-1  │──→│ B-2  │──→│ B-4  │
  └──────┘   └──────┘   └──────┘
                         ┌──────┐
                         │ B-5  │  ← B-4和B-5可以并行
                         └──────┘
             ┌──────┐
             │ B-3  │  ← B-3和B-2可以并行（同列）
             └──────┘
```

### 卡片状态

每个卡片（无论第一层还是第二层）显示：
- 名称
- 状态图标：✅完成 / 🔄进行中 / ⏳待开始 / ⚠️有冲突
- 第一层额外显示：子任务完成数（如 2/5）
- 第二层额外显示：简要描述

### 两层设计的好处

| 优势 | 说明 |
|------|------|
| 一眼看全局 | 第一层看到整个项目有几个大步骤，哪些完成了 |
| 聚焦当前 | 点击当前模块，第二层只看这个模块的子任务 |
| 依赖关系清晰 | 从左到右就是顺序，同列就是可并行，不需要额外学习 |
| 未来可扩展三层 | 架构上预留，第二层的卡片点击后可以展开第三层 |

   - 每一步选择记录在decisions/目录
   - 最终生成模块级计划文件（.tmplan/modules/）

2. **项目看板**
   - 顶部项目总进度
   - 模块卡片（显示完成度）
   - 点击展开查看具体任务列表
   - 任务状态切换（待开始/进行中/完成）

   - 偏差时给出选择题让用户决策

   - 生成新模块的计划文件，插入看板

   - 决策记录追溯
   - 计划文件版本跟随Git管理

## 九、总结

TmPlan的本质是一个**AI计划文件生成器**：

- **输入**：开发者的模糊想法
- **过程**：AI通过选择题+对话逐步引导，每一步用户确认
- **输出**：颗粒度极细的 `.tmplan/` 计划文件，任何AI模型拿到就能执行
- **持续**：自动检测计划与代码的偏差，引导用户做出调整决策

它不碰代码，只读代码。它不替用户做决定，只给选择题。
它是开发者和AI编码工具之间的"计划层"——让AI每次都知道该做什么。
